## Arquitetura Proposta (Fase 1)
1. **Identity Provider**
   - Supabase Auth como provedor único na fase inicial.
   - Método inicial: email/senha (opcionalmente magic link em fase posterior).

2. **Session Model**
   - Sessão baseada em cookie seguro, validada no servidor.
   - Refresh token nunca exposto em localStorage.
   - Expiração curta com renovação controlada.

3. **Trust Boundaries**
   - Cliente web: não confiável.
   - Next.js API/routes server: confiável com validação explícita por request.
   - Supabase: fonte de identidade e autorização de dados.

4. **Access Control**
   - Default-deny para páginas privadas e APIs sensíveis.
   - Perfis mínimos: `admin`, `analyst`, `viewer`.
   - Escopos por rota/API mapeados antes da abertura de acesso.

## Threat Model (Prioridade)
1. **Acesso não autenticado por URL direta**  
   - Risco: exposição de dados e ações sem identidade.
   - Controle: middleware obrigatório + checks server-side por rota.

2. **Token/session hijacking**  
   - Risco: sequestro de sessão por XSS ou transporte inseguro.
   - Controle: cookies `httpOnly`, `secure`, `sameSite`, rotação e expiração.

3. **Broken access control em APIs**  
   - Risco: endpoint acessível sem autorização de perfil.
   - Controle: autorização por escopo/perfil em todas as APIs sensíveis.

4. **Exposure of secrets in client bundle**  
   - Risco: vazamento de credenciais de serviço.
   - Controle: service keys apenas server-side; varredura de env exposta.

5. **Brute force / credential stuffing**  
   - Risco: tomada de conta por tentativas repetidas.
   - Controle: rate limit por IP/usuário e lockout temporário progressivo.

6. **Session fixation / stale sessions**  
   - Risco: reaproveitamento de sessão antiga.
   - Controle: invalidar sessão no login/logout e em troca de privilégio.

## Security Controls Obrigatórios
- Guard de autenticação em:
  - `/dashboard`, `/performance`, `/campaigns`, `/ads`, `/adsets`, `/leads`, `/settings`
- Guard de autenticação/autorização nas APIs:
  - todas as rotas de leitura/escrita de dados de negócio
- Cabeçalhos mínimos:
  - `X-Frame-Options: DENY`
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - CSP progressiva (report-only -> enforce)
- Logging de eventos:
  - login sucesso/falha
  - logout
  - acesso negado (rota/API)
  - sessão expirada/revogada

## Rollout Seguro
1. Feature flag para ativar auth por ambiente.
2. Deploy em staging com usuários de teste.
3. E2E obrigatório (task 30-7) antes de produção.
4. Janela de observação pós-deploy (logs/auth errors).
5. Plano de rollback sem remover validação de API crítica.
