## Onde falta detalhe — recomendações técnicas (com breve argumento)

### 1) “TimescaleDB” vs o que faremos no Supabase
- **Situação:** O repositório já tem **`adset_insights`** com chave **`UNIQUE (adset_id, date)`** e índice `(adset_id, date)` — ou seja, **já é uma série temporal diária** adequada como base do MVP.
- **Recomendação:** **Não** introduzir serviço novo (Timescale Cloud) nem depender de extensão Timescale no Postgres até haver necessidade comprovada (volume, consultas analíticas pesadas).
- **Decisão proposta para 27-1:** Tratar **`adset_insights` como fonte canônica** do grão diário por adset; a migração desta task deve **documentar o contrato** e, se útil, expor uma **view estável** para a 27-2 (colunas explícitas + comentários), em vez de duplicar dados em outra tabela sem motivo.

### 2) Pergunta 2 (“primeiro consumidor”) — tradução direta
- **Significa:** a 27-1 não precisa entregar UI nem API pública nova; precisa deixar o **dado consultável de forma previsível** para um script/serviço da **27-2** (ex.: exportar série por `adset_id` e janela de datas).

### 3) Dias sem dados (sua resposta “o que for melhor”)
- **Recomendação:** Manter **esparso** no banco (sem linha = sem dado naquele dia), que é o comportamento natural da tabela quando não há sync.
- **Por quê:** Duplicar “linha com zero” para todos os adsets × todos os dias **infla** storage e manutenção.
- **Onde tratar zeros:** Na **27-2** (preparação do dataset: preencher zeros só na janela de treino quando o modelo exigir série contínua). Assim a 27-1 fica simples e correta.

### 4) Backfill / idempotência (“não sei dizer”)
- **Situação:** `INSERT ... ON CONFLICT (adset_id, date) DO UPDATE` (ou equivalente no job de sync) já é o padrão mental para **não duplicar** dia.
- **Recomendação:** Para qualquer objeto **novo** (ex.: view materializada futura), seguir a mesma filosofia: chaves naturais claras e **rebuild idempotente** por intervalo de datas.

### 5) Pergunta 15 — interpretação de “sim”
- **Ambiguidade:** “Sim” pode ser “sim, pode ter nomes” ou leitura errada da pergunta.
- **Recomendação (segurança + simplicidade):** **Não criar cópia nova de nomes** só para ML. Usar **`adset_id`** como chave; **nomes** vêm de **`adset_insights.adset_name` / join `adsets`** quando a 27-2 precisar de rótulo para explicabilidade. Isso reduz divergência (nome mudou em um lugar só).

### 6) RLS / “mesma lógica das outras seções”
- **Situação no repo:** `adset_insights` tem **RLS habilitado** (migração de endurecimento); o app costuma acessar dados sensíveis via **server** com cliente privilegiado, não expor tudo ao browser.
- **Recomendação:** Qualquer view nova para ML deve ser avaliada com **PostgREST + RLS** em mente: se a leitura for **só backend**, pode não precisar de política ampla; se um dia for **lida no client**, aí sim políticas explícitas como nas tabelas de dashboard.

### 7) Definition of Done (sua incerteza no item 18)
- **Proposta objetiva para fechar 27-1:**
  1. Este documento com **decisões fechadas** e **contrato de dados** (seção abaixo).
  2. **Migração** em `supabase/migrations/` aplicável em dev (view e/ou comentários + ajuste mínimo de índice se justificado).
  3. **Query de smoke** documentada (neste arquivo ou em comentário SQL) que um humano roda e valida série por `adset_id` em uma janela de ~90 dias e opcionalmente ~365 dias (nice-to-have).

## Decisões fechadas para execução (até segunda ordem)

1. **Plataforma:** Supabase Postgres **somente**; sem novo banco pago na 27-1.
2. **Fonte canônica do grão diário:** `public.adset_insights` (`adset_id`, `date`, métricas já existentes, `UNIQUE(adset_id, date)`).
3. **Escopo da migração desta task:** contrato documentado + **view de leitura** para a 27-2 (nome a definir na implementação, ex.: `v_ml_adset_daily_series`) **ou** apenas `COMMENT`/documentação se a view for considerada risco RLS — **preferência técnica: view com `security_invoker`** quando suportado, para respeitar RLS das tabelas base.
4. **Histórico ~1 ano:** requisito **analítico** (consultas devem suportar janelas longas); não obriga backfill de dados que nunca existiu na origem.
5. **Fuso:** métricas por **`date` civil** alinhadas ao uso atual do produto em **America/Sao_Paulo** (documentar que `date` não carrega fuso; o boundary do dia é responsabilidade do pipeline que grava em `adset_insights`).
6. **Pipeline de carga:** para a 27-1, **reutiliza** o pipeline que já alimenta `adset_insights`; **não** exige novo scheduler nesta task salvo descobrirmos gap objetivo na implementação.
