## Onde faltava detalhe — recomendações técnicas (registro)

### 1) “TimescaleDB” vs o que faremos no Supabase
- **Situação:** O repositório já tem **`adset_insights`** com chave **`UNIQUE (adset_id, date)`** e índice `(adset_id, date)` — ou seja, **já é uma série temporal diária** adequada como base do MVP.
- **Decisão:** **Não** introduzir serviço novo (Timescale Cloud) nem depender de extensão Timescale no Postgres até haver necessidade comprovada.
- **Execução:** View estável `public.v_ml_adset_daily_series` sobre `adset_insights` (migração `20260420140000_27_1_ml_adset_daily_series_view.sql`).

### 2) Dias sem dados
- **Decisão:** Manter **esparso** no banco (sem linha = sem dado naquele dia).
- **27-2:** imputar zeros só na preparação do dataset, se o modelo exigir série contínua.

### 3) Backfill / idempotência
- **Base:** `UNIQUE (adset_id, date)` em `adset_insights`; cargas devem usar **UPSERT** por esse par natural.
- **View:** somente leitura; idempotência de dados continua responsabilidade do pipeline que grava em `adset_insights`.

### 4) RLS / `security_invoker`
- **View:** criada com **`WITH (security_invoker = true)`** (PostgreSQL 15+): avalia permissões e RLS **como o usuário que consulta**, não como dono da view.
- **Consumo previsto:** APIs Next com **`service_role`** (bypass RLS). `GRANT SELECT` na view foi concedido apenas a **`service_role`** para não ampliar superfície a `anon`.

## Queries de smoke (rodar após `supabase db push` / aplicar migração)

Substitua `'<SUBSTITUIR_ADSET_ID>'` por um `adset_id` real existente em `adset_insights` (mantenha as aspas simples se o ID for texto).

**1) Janela ~90 dias — contagem e min/max de datas**

```sql
SELECT
  count(*) AS rows_90d,
  min(metric_date) AS d0,
  max(metric_date) AS d1
FROM public.v_ml_adset_daily_series
WHERE adset_id = '<SUBSTITUIR_ADSET_ID>'
  AND metric_date >= (current_date - interval '90 days')::date
  AND metric_date <= current_date;
```

**2) Janela ~365 dias (nice to have)** — mesmo padrão com `interval '365 days'`.

**3) Esparsidade** — dias no calendário vs linhas (esperado: calendário ≥ linhas)

```sql
SELECT
  (current_date - (current_date - interval '90 days')::date) AS calendar_days_approx,
  count(*) AS row_days
FROM public.v_ml_adset_daily_series
WHERE adset_id = '<SUBSTITUIR_ADSET_ID>'
  AND metric_date >= (current_date - interval '90 days')::date;
```
