# 33-9 Hardening de sessão e middleware para rotas protegidas

[Voltar para lista de tarefas](./tasks.md)

## Description
Endurecer o controle de autenticação no middleware para evitar aceitação de cookie inválido como sessão válida e reduzir o efeito de bounce para `/login` quando há sessão renovável via refresh token.

## Requirements
- [x] Não aceitar apenas presença de cookie como autenticação válida em rotas protegidas.
- [x] Validar access token com Supabase no middleware para páginas e APIs protegidas.
- [x] Tentar refresh server-side quando access token estiver ausente/inválido e refresh token existir.
- [x] Manter redirecionamento para `/login` somente quando não houver sessão recuperável.
- [x] Preservar rastreabilidade via `logAuthAudit`.

## Implementation Plan
1. Criar validação real de access token no middleware usando `supabase.auth.getUser`.
2. Adicionar fluxo de refresh por `refreshSession` no middleware para recuperar sessão sem bounce.
3. Aplicar cookies renovados no `NextResponse` para requests de página e API.
4. Ajustar login para respeitar `redirect` de origem após `session check` e submit.

## Test Plan
1. Com `access_token` válido, acessar rota privada (`/dashboard`) e confirmar acesso sem redirecionamento.
2. Com `access_token` expirado e `refresh_token` válido, acessar rota privada:
   - não deve permanecer em `/login`;
   - sessão deve ser renovada e navegação seguir normalmente.
3. Com cookies inválidos/ausentes, acessar rota privada e confirmar redirecionamento para `/login?redirect=...`.
4. Acessar API protegida (`/api/performance`) com token inválido:
   - deve retornar `401` se refresh falhar;
   - deve seguir com sucesso se refresh for válido.
