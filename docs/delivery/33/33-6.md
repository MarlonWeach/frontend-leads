# 33-6 Expor dados de metas para consumo pelo PBI 32

## Description
Publicar um contrato de dados estável para o PBI 32 consumir metas por adset/competência, incluindo referência de entrega (manual ou fallback), gap, ritmo diário necessário e métricas de eficiência.

## Endpoint Publicado
- `GET /api/goals/optimization-context`

### Query params
- `meta_account_id` (opcional, default `256925527`)
- `competence_month` (opcional, formato `YYYY-MM`)

### Resposta por item
- `meta_account_id`
- `adset_id`, `adset_name`
- `competence_month`
- `contract_start_date`, `competence_end_date`
- `volume_contracted`
- `delivered_reference`
- `delivered_source` (`manual_client` | `historical_fallback`)
- `volume_remaining`
- `days_remaining`
- `leads_needed_daily`
- `cpl_target`
- `current_cpl`
- `budget_total`
- `spend_to_date`
- `has_manual_delivery`

## Regras de Negócio Embutidas
- Referência de entrega:
  - prioriza `volume_captured` manual;
  - fallback para histórico de `adset_insights`.
- Competência:
  - inicia em `contract_start_date`;
  - termina no último dia do mês da competência.
- Elegibilidade dinâmica:
  - disponível via `volume_remaining` e `leads_needed_daily`.

## Test Plan
Objetivo:
- Garantir que o PBI 32 consiga consumir metas de forma consistente para priorização.

Cenários:
- Chamada sem parâmetros retorna contexto da conta padrão.
- Chamada com `competence_month` filtra a competência solicitada.
- Adset com `volume_captured` manual retorna `delivered_source=manual_client`.
- Adset sem manual retorna `delivered_source=historical_fallback`.

Critério de aprovação:
- Time do PBI 32 consegue usar o endpoint sem transformar payload adicionalmente.
