## Tela 1 — Lista de recomendações

### Cabeçalho da página

- Título: **Recomendações**
- Subtítulo: **Sugestões com base nos últimos 7 dias e nas metas do período. Aplicação manual — nada é alterado na Meta sem sua confirmação.**
- Chips de filtro rápido: **Ativas** (default) | **Todas** | **Histórico do adset** (abre filtro `entity_id` avançado na 32-7)

### Filtros (barra secundária)

| Controle | Fonte API |
|----------|-----------|
| Status | Query `status` (default `active`); multiselect mapeado para CSV |
| Mês de competência | `competence_month` |
| Busca por adset | Campo texto → filtra client-side por `entity_id` / nome se API list devolver nome; ou endpoint history na 32-7 |

### Ordenação (MVP)

1. `confidence_level` (high → medium → low)  
2. Dentro do mesmo nível: `generated_at` desc  

*(Prioridade “operacional” do PRD = combinação de confiança + proximidade de expiração: opcional fase 2 — `expires_at` asc como tie-breaker.)*

### Cartão / linha de recomendação (lista)

Cada item é um **card** (padrão glass/cards do app), clicável para abrir detalhe.

**Conteúdo visível:**

| Zona | Conteúdo |
|------|------------|
| Topo esquerdo | Badge do tipo (ver legenda abaixo) + **ID curto** do adset ou nome se disponível |
| Topo direito | Pill de confiança: **Alta** / **Média** / **Baixa** (cores: verde / âmbar / cinza) |
| Corpo | `evidence_summary` (máx. 2 linhas; truncar com “Ver mais” no detalhe) |
| Rodapé esquerdo | `generated_at` formatado + “Expira em …” relativo a `expires_at` |
| Rodapé direito | Ações rápidas: **Aplicar** · **Descartar** · **Adiar** (ícones + tooltip; no mobile empilhar) |

**Badges de tipo (`recommendation_type`):**

| Valor API | Label UI |
|-----------|----------|
| `budget_increase` | Aumentar orçamento |
| `budget_decrease` | Reduzir orçamento |
| `schedule_shift` | Ajustar horários |
| `focus_adset` | Foco neste adset |

**Estados visuais na lista:**

- `active`: card cheio; ações habilitadas se `now < expires_at`.
- `expired`: opacidade reduzida; badge **Expirada**; sem ações de apply/discard; botão **Ver** só leitura.
- `applied` / `discarded`: fora do filtro default “Ativas”; no filtro “Todas”, mostrar badge **Aplicada** ou **Descartada** e data da decisão (campo agregado na 32-7 via join ou denormalização leve).

## Tela 2 — Detalhe (drawer ou modal largo)

Abre ao clicar no card (ou em **Ver**).

### Seções (colapsáveis / accordion)

1. **Resumo da ação** — tipo, `action_payload` legível (ex.: “+10% no orçamento diário, até +20% por política”).
2. **Por que recomendamos** — `score_breakdown` em micro-barras ou tabela: CPL, eficiência, volume com pesos.
3. **Metas do período** — bloco derivado de `context_snapshot.goal_context` (volume contratado, entregue, gap, CPL alvo vs atual).
4. **Últimos 7 dias** — `context_snapshot.window_7d` (datas, spend, leads, CPL, CTR, dias com dado válido).
5. **Riscos e alertas** — `risk_flags` como lista; se vazio, texto “Nenhum alerta de risco registrado.”
6. **Auditoria** — timeline curta: geração + cada decisão `defer` + decisão final (dados de `optimization_recommendation_decisions` na 32-7).

### CTAs no rodapé do detalhe

Mesmos três botões primários: **Aplicar**, **Descartar**, **Adiar** (desabilitar se status ≠ `active` ou expirado).

## Fluxos de decisão

### Aplicar

1. Clique em **Aplicar** → modal de confirmação.
2. Texto: **Confirma aplicar esta sugestão? O sistema registrará a decisão; a alteração na Meta será feita na etapa seguinte (fora do escopo automático do MVP) ou manualmente conforme processo interno.** *(Ajustar quando 32-7 integrar Meta ou só log.)*
3. Campo **Motivo (opcional)** — select `reason_code` + `note` opcional.
4. Confirmar → `POST .../decision` `{ action: "apply", ... }` → toast sucesso; card atualiza para **Aplicada** ou some do filtro Ativas.

### Descartar

1. Modal: **Por que está descartando?** — `reason_code` recomendado obrigatório (select); `note` obrigatório se `reason_code === other` (validação client + server 422).
2. Confirmar → `discard` → toast.

### Adiar

1. Modal leve: **Vamos manter esta recomendação ativa para você revisar depois.** Sem data no MVP (alinhado à 32-3).
2. `defer` → toast “Registrado”; card permanece na lista **Ativas**; linha extra na timeline de auditoria.

### Erros da API

- **409** `RECOMMENDATION_ALREADY_DECIDED`: toast + fechar modal + refetch lista.
- **409** `RECOMMENDATION_EXPIRED`: mensagem clara + refetch.
- **403**: “Sem permissão para esta conta Meta.”

## Estados vazios e carregamento

| Situação | UI |
|----------|-----|
| Loading inicial | Skeleton de 3 cards |

| Erro de rede | Banner com retry |
| Conta não selecionada (futuro) | Empty state pedindo seleção |

## Acessibilidade (mínimo MVP)

- Botões com `aria-label` quando só ícone.
- Modal: foco preso, `Esc` fecha, retorno de foco ao elemento que abriu.
- Pills de confiança com texto, não só cor.
- Toasts anunciados de forma não exclusiva de leitor de tela (padrão do design system do projeto).
