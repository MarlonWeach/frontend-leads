## Variáveis Vercel — o que manter, apagar e atualizar

Atualiza os valores para o **projeto Supabase novo** e tokens Meta válidos. Ordem sugerida: no Supabase Dashboard → **Settings → API** copia URL, anon e service_role; na Meta, token de sistema com escopos necessários.

### Apagar (recomendado)

| Variável | Motivo |
|----------|--------|
| **`NEXT_PUBLIC_META_ACCESS_TOKEN`** | **Crítico:** qualquer `NEXT_PUBLIC_*` vai parar ao bundle do browser. Token da Meta não pode ser público. O código já usa `META_ACCESS_TOKEN` (servidor) com fallback — basta manter **só** a variável sem `NEXT_PUBLIC_`. |
| **`NEXT_PUBLIC_META_APP_ID`** | Não há referências no código (`grep` no repo). Podes remover para reduzir ruído; readicionas se um feature precisar. |

### Manter (servidor — sem `NEXT_PUBLIC_` para secrets)

| Variável | Notas |
|----------|--------|
| **`SUPABASE_SERVICE_ROLE_KEY`** | Obrigatória nas API routes / servidor. Atualizar com a key do projeto **novo**. |
| **`SUPABASE_URL`** | Muitos scripts aceitam `NEXT_PUBLIC_SUPABASE_URL \|\| SUPABASE_URL` — mantém com o **mesmo** URL do projeto novo (pode ser igual ao domínio em Settings → API). |
| **`META_ACCESS_TOKEN`** | Token Meta (servidor). Atualizar. |
| **`META_ACCOUNT_ID`** | ID da conta de anúncios (`act_...`). Atualizar. |
| **`META_FORM_ID`** | IDs de formulários Lead Gen. Atualizar se mudou. |
| **`OPENAI_API_KEY`** | Servidor. Atualizar se rotacionada. |

### Manter (públicas / cliente — seguras para `NEXT_PUBLIC_`)

| Variável | Notas |
|----------|--------|
| **`NEXT_PUBLIC_SUPABASE_URL`** | URL do projeto — é pública por natureza. Atualizar. |
| **`NEXT_PUBLIC_SUPABASE_ANON_KEY`** | Chave **anon** (protegida por RLS no ideal). Atualizar. |
| **`NEXT_PUBLIC_META_ACCOUNT_ID`** | ID de conta é considerado menos sensível que o token; o código usa `NEXT_PUBLIC_META_ACCOUNT_ID \|\| META_ACCOUNT_ID`. Podes **manter** só esta **ou** só `META_ACCOUNT_ID` se não houver código cliente que precise do ID — evita duplicar o mesmo valor em duas variáveis. |

### Duplicatas a alinhar

- **`SUPABASE_URL`** vs **`NEXT_PUBLIC_SUPABASE_URL`:** devem apontar para o **mesmo** projeto. Se forem iguais, mantém as duas (vários ficheiros usam um ou outro) ou unifica no código numa tarefa futura.
- **`META_ACCOUNT_ID`** vs **`NEXT_PUBLIC_META_ACCOUNT_ID`:** se o valor for o mesmo, basta **uma** no Vercel mais a outra só se o front precisar mesmo do `NEXT_PUBLIC_`.

### Depois de editar no Vercel

1. **Redeploy** do projeto (Deployments → Redeploy) para aplicar env novas.  
2. Smoke test: homepage, uma API que use Meta, uma que use Supabase.

## Como conferir cada item (passo a passo)

### 1) URLs do projeto e **service_role** só no servidor (nunca no browser)

**No Supabase:** **Project Settings** → **API** — Project URL, anon, service_role.

**Provas no browser não encontrar secrets:** ok conforme registo acima.

**Vercel:** não uses `NEXT_PUBLIC_*` para **service_role** nem para **token Meta**. Lista acima remove `NEXT_PUBLIC_META_ACCESS_TOKEN`.

**Nota CI/scripts:** GitHub Actions pode usar nomes `NEXT_PUBLIC_*` só no ambiente do runner; o risco é **Vercel** injetar isso no Next client.

### Onde está a connection string no Supabase?

A UI muda. Experimenta por esta ordem:

1. No projeto, botão **Connect** (por vezes no topo ou ao abrir **SQL Editor** / **Database**) — abre o assistente com **URI**, **pooler** (session vs transaction), e snippets.
2. **Project Settings** → **API**: tens **Project URL**; a connection Postgres costuma ser derivável (host tipo `db.<ref>.supabase.co`) — documentação oficial: [Connecting to Postgres](https://supabase.com/docs/guides/database/connecting-to-postgres).
3. Se não aparecer “Database” dentro de **Project Settings**, é limitação da versão do painel: para **esta auditoria** basta confirmar que **Settings → API** tem URL e keys atualizadas; connection direta só é necessária para ferramentas externas (pgAdmin, etc.).

## SQL sugerido (SQL Editor)

```sql
SELECT c.relname, c.relrowsecurity
FROM pg_class c
JOIN pg_namespace n ON n.oid = c.relnamespace
WHERE n.nspname = 'public' AND c.relkind = 'r'
ORDER BY c.relrowsecurity, c.relname;

SELECT schemaname, tablename, policyname, permissive, roles, cmd, qual
FROM pg_policies
WHERE schemaname = 'public'
ORDER BY tablename, policyname;
```
