## Como conferir cada item (passo a passo)

### 1) URLs do projeto e **service_role** só no servidor (nunca no browser)

**No Supabase (só para copiar URLs e saber qual chave é qual):**

1. Dashboard → **Project Settings** (ícone de engrenagem) → **API**.
2. Ali vês **Project URL**, **anon public** e **service_role** (secret). A service_role **não** deve ser colada em ficheiros que vão para o frontend.

**O que prova que o browser não recebe a service_role:**

| Onde | O que fazer |
|------|-------------|
| **Código** | No repo, a service_role deve aparecer como `SUPABASE_SERVICE_ROLE_KEY` ou `SUPABASE_SERVICE_ROLE_KEY` **sem** prefixo `NEXT_PUBLIC_`. Tudo com `NEXT_PUBLIC_` é embutido no JavaScript do browser. |
| **Vercel** | Project → **Settings → Environment Variables**: confirma que **não** existe `NEXT_PUBLIC_SUPABASE_SERVICE_ROLE_KEY` (nem variável pública com o valor da secret). |
| **Browser (teste rápido)** | Abre o site em produção → F12 → **Sources** ou **Rede** → procura por `service_role` ou pelo início da JWT (`eyJ...`) em ficheiros `.js` servidos ao cliente. Não deve aparecer a secret longa da service role. |

**Resumo:** “Só backend/CI/Vercel” = a chave **service_role** só em variáveis **server-side** (API routes, scripts, GitHub Secrets), nunca em `NEXT_PUBLIC_*`.

**Nota sobre o repositório:** alguns **scripts Node** e o **GitHub Actions** usam o nome `NEXT_PUBLIC_SUPABASE_SERVICE_ROLE_KEY` só para ler a mesma secret em ambiente de **servidor** (CI não envia isso ao browser do utilizador). O risco é **Vercel**: aí **não** deves criar variável `NEXT_PUBLIC_*` com a service role — só `SUPABASE_SERVICE_ROLE_KEY` em ambiente server. O build Next.js só injeta `NEXT_PUBLIC_*` no bundle cliente.

### 2) Authentication — “Providers fazem sentido” e login de utilizador

**No dashboard:** menu lateral **Authentication** → **Providers**.

- Se **não** usam login Supabase para utilizadores finais (dashboard só via deploy protegido / outra auth): não precisas de ativar Google, etc. O importante é **registar a decisão** (por exemplo uma linha no `security-assessment` ou aqui): *“Sem Auth Supabase para end-users; dados via APIs Next com service role.”*
- Se **há** utilizadores: revisa quais providers estão **Enabled** e se fazem sentido.

**Checklist:** podes marcar como feito quando essa frase estiver escrita no doc de assessment ou abaixo como nota.

---

### 3) Database — onde ver no painel

| O que queres ver | Onde no Supabase |
|------------------|------------------|
| Tabelas, SQL, extensões | Menu lateral **Database** → **Tables**, **SQL Editor**, **Extensions**. |
| Connection string, pooler, parâmetros | **Project Settings** → **Database** (URL de ligação, modo session/transaction pooler, etc.). |
| Extensões instaladas | **Database** → **Extensions**. |

Não há um único “Database” escondido: é o ícone **Database** à esquerda + **Settings → Database** para ligações.

---

### 4) Network (restrições IP)

Só em **alguns planos** (ex.: Pro+ com add-on). No **Free** muitas vezes **não aparece** ou está desativado — podes marcar o item como **N/A (plano Free)** e uma nota.

---

### 5) Backups / PITR no plano **Free**

- **PITR** e backups longos costumam ser **pagos**. No Free não tens de “configurar” o mesmo menu que em Pro.
- **O que fazer na auditoria:** anotar *“Plano Free: sem PITR; aceite risco ou export periódico manual / upgrade futuro.”* e marcar o item como revisto.

## SQL sugerido (SQL Editor)

```sql
-- Tabelas public: RLS ligado ou não
SELECT c.relname, c.relrowsecurity
FROM pg_class c
JOIN pg_namespace n ON n.oid = c.relnamespace
WHERE n.nspname = 'public' AND c.relkind = 'r'
ORDER BY c.relrowsecurity, c.relname;

-- Políticas existentes
SELECT schemaname, tablename, policyname, permissive, roles, cmd, qual
FROM pg_policies
WHERE schemaname = 'public'
ORDER BY tablename, policyname;
```
